# skill-creator、prompt-design、agent-workflow-designの役割と使い分け

> **Notice：** 本文書では、2026年7月時点で筆者がCodex環境で利用しているスキルの役割を整理しています。`skill-creator`は同環境に含まれるシステムスキル、`prompt-design`と`agent-workflow-design`は独自に作成したスキルです。後者の2つはいずれもOpenAIの公式ドキュメントを参考にしており、`prompt-design`はGoogle、`agent-workflow-design`はAnthropicの公式資料も参照しています。ただし、OpenAIが提供する公式スキルではなく、ここで示す分類や使い分けもOpenAI公式の標準構成ではありません。
>
> `agent-workflow-design`は作成後の初期段階にあるため、実運用へ適用する場合は、代表的なタスクで停止条件、回復、権限、承認境界などを確認してください。

`skill-creator`、`prompt-design`、`agent-workflow-design`は、いずれもエージェント向けの仕組みを設計するときに使えますが、主に扱うレイヤーが異なります。3つを毎回すべて使うのではなく、成果物がCodexスキルかどうか、モデル向けの指示文を含むか、作業が反復・長期実行を伴うかを見て選びます。

端的には、`skill-creator`は**Codexスキルの構造・作成・検証**、`prompt-design`は**モデルに渡す指示**、`agent-workflow-design`は**処理を実行・反復・停止・回復する仕組み**を扱います。1つのCodexスキルに3つの観点が必要になることもありますが、それぞれの責務を混ぜないことが重要です。

## skill-creator：Codexスキルの構造を作る

`skill-creator`は、Codexスキルを新規作成したり、既存スキルの構造を変更したりするときに主に使います。スキル名、ディレクトリ構成、`SKILL.md`先頭のYAML frontmatter、発火条件となる`description`、`agents/openai.yaml`、`scripts/`、`references/`、`assets/`の使い分け、初期化、検証、実利用後の改善までを扱います。

中心となる考え方は、Codexがすでに持つ一般能力を前提に、繰り返し必要になる専門知識、手順、ツール連携、再利用資源だけをスキルへ持たせることです。詳細を参照ファイルへ分けて必要なときだけ読む段階的開示や、作業の壊れやすさに応じて、文章による指示、疑似コード、実行手順を固定したスクリプトを使い分ける設計も、このスキルで扱います。

一方、`skill-creator`だけで、モデル向け指示文の明確さや、長期実行ワークフローの状態遷移まで十分に設計できるとは限りません。`SKILL.md`や`description`の内容を調整する場合は`prompt-design`を、スキルで定義する作業が反復処理や中断・再開を伴う場合は`agent-workflow-design`を組み合わせます。

## prompt-design：モデル向けの指示を設計する

`prompt-design`は、プロンプト、`system`／`developer`／`agent`／`tool`向けの指示、プロンプトテンプレート、評価用プロンプト、`AGENTS.md`、Codexスキルの`description`や本文など、**モデルが読み、判断や出力に使う指示文**を設計・レビュー・修正するときに使います。Geminiの通常プロンプト、Gemの指示、画像・動画生成プロンプトも対象に含みます。OpenAIやCodex向けの成果物では、別の対象が指定されない限り、現在のGPT-5.6向けガイダンスを既定として扱います。

このスキルでは、期待する結果、必要な文脈、制約、根拠、出力形式、承認境界、停止・確認条件を整理します。重複や矛盾、曖昧な指示、過剰な手順指定を減らし、モデルに任せる判断と、ランタイムやスキーマ、権限設定などで強制する内容を分けます。既存の意図や必要な手順を保ちながら、挙動を変える情報だけを残すことを重視します。

ただし、`prompt-design`はCodexスキルのディレクトリ構成や検証手順を決める主スキルではなく、スケジュール、永続状態、再試行カウンター、排他制御などをプロンプトだけで実現するためのスキルでもありません。Codexスキルとして実装するなら`skill-creator`を、実行制御を設計するなら`agent-workflow-design`を使います。

## agent-workflow-design：反復・長期実行の制御を設計する

`agent-workflow-design`は、処理を繰り返す、長時間実行する、スケジュールやイベントに反応する、複数のツールやサブエージェントを調整する、中断から再開する、実行結果をもとに改善するといったワークフローを設計・レビュー・修正するときに使います。

対象は、トリガー、状態、処理、観察、根拠、評価、未解決差分、再試行・再計画、停止条件、回復、監査、権限、人間の承認境界です。すべてをプロンプトへ書くのではなく、意味判断はモデル向け指示、状態遷移や再試行はランタイム、進捗は永続状態、機械的に判定できる検証は評価器に分けます。公開、デプロイ、購入、削除などの操作は、人間の確認または保護された承認手順に委ねます。

このスキルは、複雑であっても1回の処理と1回の検証で終わる作業には原則として使いません。各反復で新しい根拠、変更された成果物、外部状態の変化を観察できる場合だけループを設け、必要を満たす最小の制御構造を選びます。通常のプログラム内ループと、指示文の言い換えだけを行う作業も対象外です。

## 3つの関係

| 判断したいこと | 主に使うスキル | 補助的に組み合わせるスキル |
| --- | --- | --- |
| Codexスキルをどのような構造で作り、検証するか | `skill-creator` | 指示文には`prompt-design`、反復実行には`agent-workflow-design` |
| モデルに何をどの強さ・順序・形式で指示するか | `prompt-design` | Codexスキル化するなら`skill-creator` |
| 作業をいつ開始し、何を保持し、何を見て続行・停止・回復するか | `agent-workflow-design` | モデル向け判断には`prompt-design`、スキル化するなら`skill-creator` |

3つの関係は、上位・下位の序列ではなく責務の分担として捉えます。`skill-creator`がスキルの入れ物と再利用資源を整え、`prompt-design`がその中のモデル向け指示を整え、`agent-workflow-design`が必要に応じてプロンプトの外側を含む実行制御を整えます。

## 選び方

依頼を受けたら、次の順で判断すると使い分けやすくなります。

1. **成果物はCodexスキルか。** 新しいスキルの作成や、既存スキルの構造変更なら、`skill-creator`を主に使います。
2. **モデル向けの指示文を作成・評価・修正するか。** 該当するなら、`prompt-design`を使います。Codexスキルの新規作成や構造変更を伴う場合は、`skill-creator`と併用します。
3. **処理が反復・長期実行・イベント駆動・中断再開・実行結果による改善を含むか。** 該当する場合だけ、`agent-workflow-design`を加えます。

たとえば、単発の専門作業を支援するCodexスキルを作る場合は、`skill-creator`を主に使い、`prompt-design`で`description`と`SKILL.md`を確認します。永続状態や再試行を追加する必要がなければ、`agent-workflow-design`は使いません。

一方、定期実行する調査エージェントを設計し、Codexスキルとして配布しない場合は、`agent-workflow-design`でトリガー、状態、評価、停止、回復を設計し、`prompt-design`でモデル向け指示を整えます。`skill-creator`は使いません。

## 3つを併用する場合の進め方

反復・長期実行する作業を定義するCodexスキルを作る場合は、次の順を基本にします。

1. `skill-creator`で利用例、発火条件、スキルの境界、必要な再利用資源を整理します。
2. `agent-workflow-design`で、ループの必要性、トリガー、状態、観察・評価、未解決差分、停止・回復・承認境界を設計します。
3. `prompt-design`で、モデルが担う判断、必要な根拠、ツール利用、出力、引き渡し条件を明確な指示文へ落とします。
4. `skill-creator`に戻り、`SKILL.md`、参照、スクリプト、メタデータの整合を確認し、スキルを検証します。
5. 代表的な依頼で実際の挙動を確認し、問題が見つかったレイヤーだけを修正します。

この順序を毎回固定する必要はありません。既存スキルの一文を直すだけなら`prompt-design`だけで足り、既存エージェントの停止条件だけを見直すなら`agent-workflow-design`が中心になります。必要なスキルだけを選び、構造、指示、実行制御のどこに問題があるかを分けて修正します。
